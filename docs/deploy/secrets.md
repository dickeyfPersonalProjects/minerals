# Secret inventory

This file lists every Kubernetes `Secret` the minerals deployment
depends on — what it's named, what's inside it, who reads it, and how
the value ends up in the cluster. Pair it with [`README.md`](./README.md)
(the base/overlay split and Flux flow) and [`encrypt.md`](./encrypt.md)
(the kubeseal mechanics for operator-sealed Secrets).

The inventory is the operator's checklist when standing up a new
environment, and the contract that any change introducing a new
secret-backed setting has to update.

---

## Inventory

| Name | Type | Keys (consumer field) | Consumer | Provenance | Per-env or shared |
|---|---|---|---|---|---|
| `minerals-s3-creds` | `Opaque` | `S3_ACCESS_KEY_ID` (env `S3_ACCESS_KEY_ID`), `S3_SECRET_ACCESS_KEY` (env `S3_SECRET_ACCESS_KEY`) | `Deployment/minerals` main container `envFrom: secretRef` ([`kustomize/base/deployment.yaml`](../../kustomize/base/deployment.yaml)) | Operator-sealed via kubeseal ([`encrypt.md`](./encrypt.md)) | Per-env (ciphertext bound to namespace) |
| `minerals-minio-config` | `Opaque` | `config.env` (shell exports of `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD`) | MinIO `Tenant` `spec.configuration.name` ([`kustomize/base/minio.yaml`](../../kustomize/base/minio.yaml)) | Operator-sealed via kubeseal ([`encrypt.md`](./encrypt.md)) | Per-env |
| `minerals-pg-app` | `Opaque` | `uri` (env `DATABASE_URL` via `secretKeyRef`), plus `username`, `password`, `host`, `port`, `dbname`, `jdbc-uri` (unused by app) | `Deployment/minerals` main + `migrate` initContainer ([`kustomize/base/deployment.yaml`](../../kustomize/base/deployment.yaml)) | **Auto-generated by CNPG** at `Cluster` initdb. Not committed; not sealed. | Per-env |
| `web-crt` | `kubernetes.io/tls` | `tls.crt`, `tls.key` | `Ingress` `spec.tls[].secretName` ([`example/staging/ingress.yaml`](./example/staging/ingress.yaml)) | **Auto-generated by cert-manager** from the per-env `Certificate` resource ([`example/staging/certificate.yaml`](./example/staging/certificate.yaml)) | Per-env |

The migrate initContainer reads `DATABASE_URL` from `minerals-pg-app`
but does **not** mount `minerals-s3-creds` — `migrate` only talks to
Postgres, so it intentionally has no S3 credentials. Subcommand-aware
config validation (resolved in `mi-dmv`) lets that work without
loosening the prod strictness check.

The app reads everything as env vars regardless of upstream source —
ConfigMap, sealed Secret, or operator-generated Secret all surface as
`os.Getenv` calls inside `internal/config/`. See
[`CONFIG.md`](../../CONFIG.md) for the settings inventory and the
prod-strictness rules.

### What is **not** in this inventory

- **Image pull secrets.** The current image registry
  (`ghcr.io/dickeyfpersonalprojects/minerals`) is public, so no
  `imagePullSecrets` Secret exists. If the registry is later made
  private, an `imagePullSecrets` entry on the Deployment's PodSpec and
  a backing Secret would be added — see `README.md` "Optional add-ons".
- **Future per-feature Secrets** (e.g. external API keys). Not
  pre-documented here. Anything added later must come in via the
  "Adding a new secret-backed setting" workflow below, which updates
  this inventory in the same PR.

## Provisioning Secrets in a new environment

When standing up `mineral-<env>`, work through this checklist.
Operator-sealed Secrets are produced once per env and committed to
fleet-infra; auto-generated Secrets are created by their respective
operators on first reconcile and never appear in any manifest.

### 1. Operator-sealed Secrets (kubeseal)

For each of the two operator-sealed Secrets, follow the kubeseal
workflow in [`encrypt.md`](./encrypt.md). The ciphertext is bound to
**namespace + name**, so each env requires its own encryption pass —
copying staging's ciphertext into a prod overlay will not decrypt.

- **`minerals-s3-creds`** — populate from a MinIO IAM user (or root)
  scoped to the `minerals` bucket. Plaintext input lives in
  `.sec/minerals-s3-creds.yaml`; commit only the encrypted output. The
  example overlay's stub is at
  [`example/staging/minerals-s3-creds.yaml`](./example/staging/minerals-s3-creds.yaml).
- **`minerals-minio-config`** — `config.env` is a shell-style file:
  ```
  export MINIO_ROOT_USER=...
  export MINIO_ROOT_PASSWORD=...
  ```
  Plaintext input in `.sec/minerals-minio-config.yaml`. Stub at
  [`example/staging/minerals-minio-config.yaml`](./example/staging/minerals-minio-config.yaml).

### 2. Auto-generated Secrets (CNPG, cert-manager)

These are not committed and not sealed — they materialize after Flux
applies the manifests that drive their controllers. Verify each with
`kubectl` after the first reconcile:

- **`minerals-pg-app`** — created by CloudNativePG when the `Cluster`
  resource (`kustomize/base/postgres.yaml`) initializes. If missing:
  ```bash
  kubectl get cluster -n mineral-<env>
  kubectl describe cluster minerals -n mineral-<env>
  kubectl logs -n cnpg-system -l app.kubernetes.io/name=cloudnative-pg
  ```
  Common causes for non-materialization: the CNPG operator isn't
  installed in the cluster; the `Cluster` resource has a validation
  error; storage class isn't bound.
- **`web-crt`** — created by cert-manager when the per-env `Certificate`
  resource succeeds. If missing:
  ```bash
  kubectl get certificate -n mineral-<env>
  kubectl describe certificate web-crt -n mineral-<env>
  kubectl get challenge,order -n mineral-<env>
  ```
  Common causes: the referenced `ClusterIssuer` doesn't exist or is
  failing DNS-01/HTTP-01 challenge; the hostname doesn't resolve to the
  cluster's ingress.

### 3. Verification

After Flux has reconciled the new env, this enumeration should match
the inventory exactly:

```bash
kubectl get secret -n mineral-<env>
```

Expected (names; types in parentheses):

- `minerals-s3-creds` (`Opaque`) — from the SealedSecret you committed
- `minerals-minio-config` (`Opaque`) — from the SealedSecret you committed
- `minerals-pg-app` (`Opaque`) — created by CNPG
- `web-crt` (`kubernetes.io/tls`) — created by cert-manager

You may additionally see CNPG-internal Secrets (e.g.
`minerals-ca`, `minerals-replication`, `minerals-server`) and
sealed-secrets controller-managed extras. Those are operator
plumbing, not inputs to this deployment, and don't appear in the
inventory.

## Adding a new secret-backed setting

When a feature needs a new credential, token, or key, follow this
workflow. It composes with — does not replace — the
[`CONFIG.md`](../../CONFIG.md) "Adding a new setting" procedure.

1. **Decide: existing Secret or new Secret?** Bias toward an existing
   one. Add a key to `minerals-s3-creds` if the value is in the same
   trust boundary (S3/MinIO credentials). Create a new Secret only
   when the consumers, rotation cadence, or trust boundary differ
   meaningfully — e.g. an external API key shouldn't share a Secret
   with the database password.
2. **Wire the consumer in `kustomize/base/`.** If the new value is read
   by the app and lives in an existing Secret, no base change is
   needed (the existing `envFrom: secretRef` already pulls every key).
   If a new Secret is introduced, add the `envFrom: secretRef` or
   `env.valueFrom.secretKeyRef` entry to the consuming workload.
   Plain ConfigMap-style additions (non-secret env vars) are fine to
   land in the base directly; secret values stay in the overlay.
3. **Update this inventory.** Add a row with all six columns filled —
   name, type, keys, consumer, provenance, per-env or shared. If a
   key was added to an existing Secret, update that row's `Keys`
   column rather than creating a new row.
4. **Update [`CONFIG.md`](../../CONFIG.md).** Per the new-setting
   workflow, add the setting to the inventory. If/when `CONFIG.md`
   distinguishes secret vs non-secret rows (today every row is
   `Kind=env`), link the secret rows back to this inventory.
5. **Update the example overlay.** If the new Secret is operator-
   sealed, add a redacted `SealedSecret` stub to both
   [`example/staging/`](./example/staging/) and
   [`example/prod/`](./example/prod/) with `REPLACE_WITH_KUBESEAL_OUTPUT`
   placeholders, mirroring the existing stubs. If the new Secret is
   operator-generated (CNPG, cert-manager, etc.), no example file is
   needed — note the provenance in the inventory row.
6. **Fleet-infra changes still need human approval.** Producing the
   real ciphertext for a new operator-sealed Secret and committing it
   to fleet-infra is a human-approved step, per
   [`encrypt.md`](./encrypt.md) "Commit rules". Polecats and other
   automated agents stop at the example stub.
